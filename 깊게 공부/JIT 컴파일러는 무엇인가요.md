## JIT 컴파일러는 무엇인가요?

Java 소스 코드는 먼저 javac 명령을 통해 바이트코드(.class) 로 변환된다.

하지만 이 바이트코드는 CPU가 직접 이해할 수 있는 기계어가 아니다. 

따라서 실제로 프로그램이 실행될 때는 JVM 내부에서 한 번 더 변환하는 과정이 필요하다.

바로 이 바이트코드 → 기계어 변환을 실행 시점에 수행하는 주체가 JIT 컴파일러다.

```java
JIT = "동작 중인 코드를 빠르게 만들기 위해, 실행 시점에 기계어로 변환해주는 기술"
```

JIT 컴파일러는 이름은 ‘컴파일러’이지만, 실행 중에 동작하는 동적 변환 기술이라는 특징을 가진다.

정적인 컴파일과 달리, 실제로 프로그램이 어떤 방식으로 실행되는지 관찰한 뒤 자주 실행되는 코드(HotSpot) 를 선별하여 최적화한다는 점이 핵심이다.


<br/>

## 컴퓨터 프로그램 실행 방식 두 가지

1. 인터프리트 방식
	•	코드를 한 줄씩 읽고 즉시 실행하는 방식
	•	변환 과정을 실행할 때마다 반복하므로 간단하지만 속도가 느리다
	•	예: Python, JavaScript(초기), 쉘 스크립트

2. 정적 컴파일 방식
	•	실행 전에 전부 기계어로 변환 → 실행
	•	변환 비용은 딱 한 번만 발생
	•	실행 속도가 매우 빠르지만, 실행 환경·플랫폼에 따라 빌드를 별도로 해야 하는 단점이 있다
	•	예: C, C++, Rust

<br/>

## 컴퓨터 프로그램 실행 방식 두 가지

그렇다면 JIT는 무엇인가?

JIT는 위 두 가지 방식을 혼합한 형태라고 보면 가장 이해하기 쉽다.
	•	기본적으로는 인터프리터처럼 바이트코드를 읽어 실행한다.
	•	하지만 실행 중 반복적으로 실행되는 코드를 탐지하면, 그 부분을 기계어로 변환하여 캐싱한다.
	•	이후부터는 인터프리트가 아닌 기계어로 바로 실행되므로 매우 빠르게 동작한다.

즉, 초기 실행은 인터프리트, 일정 시간이 지나면 컴파일된 기계어로 여유 있게 가속되는 구조다.

```java
인터프리트  +  런타임 컴파일(최적화)  = JIT
```

<br/>

## JIT 컴파일러의 장점

1. 실행 속도 향상

인터프리트 실행보다 기계어 실행이 훨씬 빠르기 때문에, 일정 시간이 지나면 정적 컴파일 언어에 가까운 속도가 나온다.

2. 런타임 정보 기반 최적화

정적 컴파일러는 실행 환경을 모른다.

하지만 JIT는 실행 중 실제 데이터를 보고 최적화를 수행하므로 더 정교한 최적화가 가능하다.

3. 플랫폼 독립성 유지

Java의 핵심 장점인 “Write Once, Run Anywhere” 를 그대로 유지하면서 속도까지 확보한다.


<br/>

## JIT 컴파일러가 없다면 어떻게 될까?

- Java는 바이트코드를 인터프리터로만 실행해야 한다 → 속도가 대폭 느려짐

- 루프 최적화, 인라이닝 같은 JVM의 고급 최적화가 작동하지 않음

- 고성능이 필요한 서버 애플리케이션, 빅데이터 엔진, 스트리밍 시스템에서 Java가 지금처럼 널리 쓰이지 못했을 가능성이 높다
